USE CASE NARRATIVE

User Management (CRUD Permissions)

Use Case ID UM-02 Version 1.0
Use Case Name CRUD Permissions Date 2024-11-02
Author Tony Power Complexity Medium
Priority High
Use Case Type:
☐ Business Requirement
☒ System Analysis
Source Business Requirement
Primary Actor Admin
Other Actor(s) None
Stakeholders
  • HR
  • IT Department
  • Management
Description Allows the Admin to manage system permissions by creating, reading, updating, and deleting permission levels. This functionality ensures that users have appropriate access rights to perform their roles securely.
Precondition(s)
  • Admin is authenticated and logged into the system.
  • Database of permissions is accessible.
Trigger(s) Admin initiates a request to create, view, edit, or delete a permission level.

Basic Flow (Typical Course of Events)

Actor Action System/Result, Info required
1. Admin selects "Manage Permissions" option 1. System displays list of all permissions
2. Admin chooses "Add New Permission" 2. System prompts for permission details (e.g., name, description)
3. Admin enters required details and saves 3. System saves new permission and confirms creation
4. Admin selects an existing permission to edit 4. System displays current permission settings
5. Admin modifies details and saves changes 5. System confirms that changes are saved
6. Admin selects "Delete" on a permission 6. System prompts for confirmation before deletion
7. Admin confirms deletion 7. System removes permission and confirms deletion

Alternate Flow

Actor Action System/Result, Info required
1. Admin attempts to delete a core system permission 1. System warns that core permissions cannot be deleted
Conclusion Admin successfully manages permission settings, ensuring secure and appropriate access control for users.
Post Condition(s) Updated permissions are stored in the database and applied to relevant users.

Additional Information

Business Rules
  • Only Admin has permission to manage system permissions.
  • Core permissions (e.g., "Admin Access") cannot be deleted.
Special Requirements Permissions should align with organizational roles and security policies.
Constraints and Specifications Only authorized users can access the permissions management section.
Technology and Data System database with role-based access control capabilities.
Frequency As needed, typically when new roles or access changes are required.
Assumptions Admin has sufficient understanding of roles and permissions within the organization.
Notes / Open Issues Determine if permissions should be grouped or tiered by department or function for scalability.
