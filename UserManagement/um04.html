USE CASE NARRATIVE

User Management (CRUD Login)

Use Case ID UM-04 Version 1.0
Use Case Name Login (Admin) Date 2024-11-02
Author Tony Power Complexity Low
Priority High
Use Case Type:
☐ Business Requirement
☒ System Analysis
Source Business Requirement
Primary Actor Admin
Other Actor(s) None
Stakeholders
  • IT Department
  • Security Team
Description This use case enables an Admin to log into the system securely, verifying credentials before granting access to system management functions.
Precondition(s)
  • Admin has valid credentials stored in the system.
  • Login page is accessible.
Trigger(s) Admin accesses the login page to begin an authenticated session.

Basic Flow (Typical Course of Events)

Actor Action System/Result, Info required
1. Admin enters username and password 1. System verifies credentials
2. Admin clicks "Login" 2. System authenticates login, granting or denying access
3. If credentials are valid 3. System grants access, Admin redirected to the dashboard
4. If credentials are invalid 4. System displays error message and requests re-entry of credentials

Alternate Flow

Actor Action System/Result, Info required
1. Admin forgets password and selects "Forgot Password" 1. System prompts for email to send reset link
Conclusion Admin is successfully authenticated and can now access system management functions if credentials are valid.
Post condition An authenticated session is established, allowing Admin access to restricted system areas.

Additional Information

Business Rules
  • Admin passwords must meet security requirements (e.g., length, complexity).
  • Failed login attempts should be limited to prevent unauthorized access.
Special Requirements System should use encryption for password storage and transmission.
Constraints and Specifications Multi-factor authentication may be required for additional security.
Technology and Data User database with encrypted credentials; authentication protocols.
Frequency Daily or as required by the Admin.
Assumptions Admin possesses the necessary credentials and is authorized for system access.
Notes / Open Issues Consider implementing an account lockout mechanism after a specified number of failed attempts.
